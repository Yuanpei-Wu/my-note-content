# OpenTelemetry Collector OOM 防护配置指南

## 对话详情

**用户:** otel collector 如何配置 receiver 和 processor，可以限制上报次数或者上报数据大小，防止oom

**助手:** 根据搜索结果，我来为你总结 OpenTelemetry Collector 中防止 OOM 的关键配置方法。

### 防止 OOM 的核心配置

#### 1. Memory Limiter Processor（最重要）

`memory_limiter` 必须是 pipeline 中的**第一个 processor**，用于监控内存使用情况并在达到阈值时拒绝新数据。

```yaml
processors:
  memory_limiter:
    check_interval: 1s           # 检查间隔
    limit_mib: 750               # 硬限制：超过此值将拒绝数据
    spike_limit_mib: 128         # 软限制：触发 GC 的阈值
    # 或者使用百分比（从 cgroups 读取容器内存限制）
    # limit_percentage: 75
    # spike_limit_percentage: 20

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, batch]  # memory_limiter 必须放第一位
      exporters: [otlp]
```

#### 2. Batch Processor（控制批次大小）

```yaml
processors:
  batch:
    timeout: 1s                  # 批处理超时时间
    send_batch_size: 1024        # 达到此数量时发送
    send_batch_max_size: 2048    # 最大批次大小（硬限制）
```

**内存优化建议**：
- 缩短 `timeout` 减少内存持有时间
- 减小 `send_batch_size` 和 `send_batch_max_size` 降低峰值内存

#### 3. Receiver 消息大小限制

```yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
        max_recv_msg_size_mib: 16      # 最大接收消息大小
        max_concurrent_streams: 100    # 最大并发流数
      http:
        endpoint: 0.0.0.0:4318
        max_request_body_size: 16777216  # 16MB，HTTP 请求体大小限制
```

#### 4. Exporter 队列限制

```yaml
exporters:
  otlp:
    endpoint: backend:4317
    sending_queue:
      enabled: true
      queue_size: 500          # 发送队列大小
      num_consumers: 5         # 消费者数量
```

#### 5. GOMEMLIMIT 环境变量

设置 Go 运行时内存限制（建议设置为容器内存限制的 80%）：

```yaml
# Kubernetes 示例
env:
  - name: GOMEMLIMIT
    value: "2048MiB"  # 设为容器内存限制的 80%
```

### 完整配置示例

```yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
        max_recv_msg_size_mib: 16
      http:
        endpoint: 0.0.0.0:4318
        max_request_body_size: 16777216

processors:
  # 1. memory_limiter 必须放在第一位
  memory_limiter:
    check_interval: 1s
    limit_mib: 1024
    spike_limit_mib: 256

  # 2. batch 控制批次大小
  batch:
    timeout: 200ms          # 短超时减少内存占用
    send_batch_size: 512    # 较小的批次
    send_batch_max_size: 1024

exporters:
  otlp:
    endpoint: backend:4317
    sending_queue:
      enabled: true
      queue_size: 500       # 限制队列大小

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, batch]  # 注意顺序
      exporters: [otlp]
```

### 关键要点

| 配置项 | 作用 | 建议值 |
|--------|------|--------|
| `memory_limiter.limit_mib` | 硬内存限制 | 容器内存的 75-80% |
| `memory_limiter.spike_limit_mib` | 软限制/GC 触发 | limit_mib 的 20-25% |
| `batch.send_batch_max_size` | 最大批次 | 根据内存调整，通常 1024-2048 |
| `batch.timeout` | 批处理等待时间 | 内存紧张时用 200ms，正常用 1s |
| `max_recv_msg_size_mib` | 单条消息上限 | 根据实际数据大小设置 |

**最佳实践**：
1. `memory_limiter` 必须作为第一个 processor
2. 同时配置 `GOMEMLIMIT` 和 `memory_limiter` 双重保护
3. 根据实际数据量和容器资源调整各参数

## `spike_limit` 的作用和工作机制

`spike_limit_mib` 或 `spike_limit_percentage` 是 memory_limiter 的**软限制/警戒水位**，用于在达到硬限制之前提前触发保护机制。

### 双阈值模型

```
内存使用量
    ↑
    │    极限 (OOM 风险区)
    │    ┌─────────────────────
    │    │  ← spike_limit 触发的区域
    │────┼───────────────────── ← limit_mib (硬限制)
    │    │
    │    │  ← spike_limit_mib (软限制) ──────┐
    │    │                                    │
    │────┼────────────────────────────────────┘ ← 开始拒绝部分数据，强制 GC
    │    │
    │    │  ← 正常工作区域
────┼────┴─────────────────────
    └──────────────────────────→ 时间
```

### 两个阈值的行为差异

| 阈值 | 参数名 | 行为 |
|------|--------|------|
| **软限制** | `spike_limit_mib` | 超过时触发 **GC** 和 **部分限流**，尝试降低内存 |
| **硬限制** | `limit_mib` | 超过时**完全拒绝**新数据，返回错误给上游 |

### 具体工作机制

```yaml
processors:
  memory_limiter:
    check_interval: 1s
    limit_mib: 1000        # 硬限制：1000 MiB
    spike_limit_mib: 200   # 软限制：200 MiB
```

**实际触发点**：

1. **软限制触发点** = `limit_mib - spike_limit_mib` = **800 MiB**
2. **硬限制触发点** = `limit_mib` = **1000 MiB**

#### 阶段一：内存 800 MiB（软限制）
- 内存使用超过 `limit_mib - spike_limit_mib` (1000 - 200 = 800)
- Collector 开始**触发 GC**
- **按比例拒绝数据**（部分限流）
- 尝试将内存压下来

#### 阶段二：内存 1000 MiB（硬限制）
- 内存使用超过 `limit_mib`
- Collector **完全拒绝**所有新数据
- 向上游返回错误（SDK 可能重试或丢弃）

### 使用百分比时的计算

```yaml
processors:
  memory_limiter:
    limit_percentage: 80          # 硬限制：总内存的 80%
    spike_limit_percentage: 25    # 软限制区间：25%
```

- **软限制触发点** = 80% - 25% = **55%**
- **硬限制触发点** = **80%**

### 配置建议

| 场景 | 推荐配置 |
|------|----------|
| 容器内存 2Gi，想保守一点 | `limit_mib: 1200` (60%)，`spike_limit_mib: 300` (15%) |
| 容器内存 2Gi，正常负载 | `limit_mib: 1500` (75%)，`spike_limit_mib: 300` (15%) |
| 使用百分比（推荐） | `limit_percentage: 75`, `spike_limit_percentage: 20` |

### 为什么要设计两个阈值？

1. **提前预警**：软限制给系统一个缓冲期，通过 GC 尝试自我恢复
2. **避免剧烈抖动**：如果没有软限制，直接从正常状态跳到完全拒绝，会导致流量剧烈波动
3. **渐进式保护**：先尝试轻量级处理（GC），不行了再完全拒绝

### 关键注意事项

- `spike_limit` 必须**小于** `limit`，否则配置会报错
- 通常 `spike_limit` 设置为 `limit` 的 **20-25%**
- 如果流量突发频繁，可以适当增大 `spike_limit` 提前触发保护
