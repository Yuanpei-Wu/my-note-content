## 写在前面

可观测性至少包括 Log、Metric、Tracing。在 Metric 方面，Prometheus 提供了一套完整的模型。它至少有如下功能：

* 收集指标 （PushGateway or Scrape）

* 把指标存储在时间序列的db中 (TSDB)

* 发送报警 (AlertManager)

* 提供 http server，让其他组件访问存储的指标

* 提供 dashboard，可以用 promQL 查询（Grafana）

![1.00](https://pub-2f2f8e6626cf43d389ec1fa8a2d85504.r2.dev/images/1778425067427-qjahga.png)

<br />

限于篇幅，本次分享主要聚焦于指标在 Prometheus 中的数据流，包括指标的抓取、存储和查询。希望本次分享能回答如下问题：

* Prometheus 如何抓取指标的？

* 指标在内存和磁盘中是如何存储的？查询时如何快速定位需要的指标并且将它们读取出来？

* 指标在上报时只是一个个散点，我们在用 promQL 查询计算后如何成为一条条观测曲线的？

* 为什么要设计 Counter Gauge Histogram Summary 这几种指标类型？这几种指标类型是不是就能够覆盖我们所有的指标观测需求了？

## 1. Scrape

### 1.1 指标采集方式

Prometheus 有一套完整的抓取方案来采集指标。

抓取指标需要回答两个问题：从哪抓，抓什么。

**从哪抓？** 即如何获得抓取目标。

* 在 k8s 集群中可以依赖服务发现（ServiceMonitor）来确定拉取目标。比如，我们可以配置抓取目标是符合某个 label 的 service 的所有 pod （IP + Port）

* 默认的路径是 /metrics，可以在 ServiceMonitor 中配置

* 抓取间隔也是固定的，也可以在 ServiceMonitor 中配置

  * 每一个抓取目标都有一个独立的计时器，这样虽然抓取间隔固定，但是服务发现时间天然错峰的，就可以避免同一时间抓取过多目标

    <br />

**抓什么？**  即应用如何暴露数据。

* 应用埋点

  * 应用可以使用 Prometheus 客户端库在 /metrics 接口暴露数据

* 安装 Exporter

  * 有很多应用都有对应的一个“指标翻译官”，将应用的指标转换为符合 Prometheus 格式的指标。例如 redis 就有一个对应的 redis-exporter

  * kube-state-metrics，通过调用 api-server 获得一些 deployment / statefulset 实例数量

  * dcgm-exporter，硬件厂商实现的 exporter，获取 GPU 利用率、功耗等

* PushGateway

  * 适用于一些主动推送指标的场景，例如一个 job 跑完后实例就退出了，没法等 Prometheus Server 来拉取，可以推送到 PushGateway，Prometheus Server 再定期从 PushGateway 拉取指标

### 1.2 指标数据格式

我们来看看暴露出来的 /metrics 接口，数据是什么样的

```

# HELP http_requests_total The total number of HTTP requests.
# TYPE http_requests_total counter
http_requests_total{method="post",code="200"} 1027 1395066363000
http_requests_total{method="post",code="400"}    3 1395066363000
# HELP http_request_duration_seconds A histogram of the request duration.
# TYPE http_request_duration_seconds histogram
http_request_duration_seconds_bucket{le="0.05"} 24054
http_request_duration_seconds_bucket{le="0.1"}  33444
http_request_duration_seconds_bucket{le="0.2"}  100392
http_request_duration_seconds_bucket{le="0.5"}  129389
http_request_duration_seconds_bucket{le="1"}    133988
http_request_duration_seconds_bucket{le="+Inf"} 144320
http_request_duration_seconds_sum 53423
http_request_duration_seconds_count 144320
# HELP cpu_temperature CPU temperature in celsius
# TYPE cpu_temperature gauge
cpu_temperature{core="0"} 65.3
```

<br />

每行一条采样数据，格式为：

```
metric_name[{label1="value1", label2="value2", ...}] value [timestamp]
```

* `metric_name`：指标名称

* `{...}`：标签键值对，可选；**同一指标名 + 不同标签组合 = 不同的时间序列**

* `value`：采样值，浮点数（支持科学计数法如 `1.027e+03`）

* `timestamp`：可选，Unix 毫秒时间戳；若省略则由 Prometheus 抓取时赋值

注释行以 `#` 开头，有两种特殊注释：

| 关键字    | 作用            | 示例                                                              |
| :----- | :------------ | :-------------------------------------------------------------- |
| `HELP` | 指标的文档说明（人类可读） | `# HELP http_requests_total The total number of HTTP requests.` |
| `TYPE` | 声明指标类型        | `# TYPE http_requests_total counter`                            |

<br />

### 1.3 关于指标类型

这里需要注意：`TYPE` 和 `HELP` 信息不进入 TSDB。

Prometheus 在抓取时解析到 `# TYPE http_requests_total counter`，知道了指标类型，但这个信息只用于：

* Prometheus 自身的 `/api/v1/metadata` 接口返回类型信息

  比如用如下指令查看某个指标的指标类型

  ```
  curl -g '112.123.118.158:32301/api/v1/metadata' | python3 -m json.tool |head -n 10

  {
      "status": "success",
      "data": {
          "DCGM_FI_DEV_DEC_UTIL": [
              {
                  "type": "gauge",
                  "unit": "",
                  "help": "Decoder utilization (in %)."
              }
          ],
          ...
      }
  }
  ```

  <br />

* 客户端库（如 Grafana）展示时的辅助信息

  ![1.00](https://pub-2f2f8e6626cf43d389ec1fa8a2d85504.r2.dev/images/1778681634860-cnv7lz.png)

  <br />

在 TSDB 中，**counter 和 histogram 本质上没有什么不同**——它们都是一组 `(labels, timestamp, value)` 的采样序列。一条 `http_requests_total{method="post",code="200"}` 的 counter 序列和一条 `http_request_duration_seconds_bucket{le="0.1"}` 的 histogram 序列，在存储层是完全同构的。

这意味着：

* **类型不约束写入行为**：你往一个 `counter` 序列写入递减值，Prometheus 不会拒绝（但 `rate()` 函数会做 counter reset 修正，详见 3.3）

* **类型不约束查询行为**：你可以对 gauge 用 `rate()`，Prometheus 不阻止（但结果可能无意义）

* **类型是"约定"而非"约束"**：类型信息只在客户端库和函数语义层面有意义，不在存储和查询引擎层面

<br />

### 1.4 改写标签

抓取指标时，Prometheus 会将指标名称转换为 `__name__ `           这个标签。所以**指标名称在 Prometheus 内部处理时只是一个标签**

此外，relabel 机制可以在抓取 metric 时对 label 追加、改写或者删除。

relabel 有如下作用：

* 加上公共的标签，例如集群名称

* 加上其它元信息，例如将 pod label/annotation 映射为 metric label

* 丢弃高基数标签，例如 user\_id

比如 dcgm-exporter 配置 relabel 如下：

```
    relabelings:
    - action: replace
      regex: (.*)
      replacement: $1
      sourceLabels:
      - __meta_kubernetes_endpoint_node_name
      targetLabel: instance
```

## 2. Store

指标采集上来后，就是一个个时间序列数据，Prometheus 将这些时间序列存储到 TSDB 中。

从结构上来看，TSDB 就是一个典型的 LSM-Tree 架构存储引擎

* 顺序写入

* 在内存保留一定大小/时间后写入到磁盘的块文件

* 块文件合并，按照体积/时间合并

* 建立索引查询

同时，TSDB 针对时间序列数据“高并发写入、按时间范围查询”的特点进行了深度优化。

<br />

### 2.1 Head Block

先看看时间序列在内存中的数据格式。为了方便理解，我们以一个具体的时间序列为例。

假设我们有两个时间序列，抓取间隔为 15 秒，观察它们在10:00-12:00 2 小时内的数据流转：

* **Series 1**: `http_requests_total{method="GET", path="/api"}`

* **Series 2**: `http_requests_total{method="POST", path="/api"}`

每个序列的数据点大致如下：

* S1: (10:00:00, 100), (10:00:15, 105), (10:00:30, 105)...

* S2: (10:00:00, 50), (10:00:15, 52), (10:00:30, 52)...

S1在内存中的结构称之为 Head Block，大概是下面这个样子

\
`[Header] [1690000000, 100.0] [1bit_DOD, 1.37byte_XOR] [1bit_DOD, 1.37byte_XOR] ...`

<br />

* **Header (4 bytes)**：`[长度][编码类型]`

* **Sample 1 (基准点，无压缩，占 16 bytes)**：

  * `Time` (8 bytes int64): `1690000000` (10:00:00 的 Unix 时间戳)

  * `Value` (8 bytes float64): `100.0`

* **Sample 2 (增量压缩，极其紧凑，可能只占 2-3 bits)**：

  * *时间戳*：和前一个时间戳差 15 秒。采用 Delta-of-Delta 编码。差值的差值为 0（因为 15-15=0），只需存一个标志位 `0`，占用 1 bit。

    * Delta-of-Delta 编码就是记录二次差值（差值的差值），在差值是固定数值时，二次差值就一直是0

  * *值*：105.0 和 100.0 的二进制异或结果。XOR 编码发现有效数字位很少，可能只需要存 1 bit 控制位 + 几位有效位。平均下来一个点只需 **1.37 bytes**。

### 2.2 Persist Block

每隔2h，内存中的 Head Block 持久化到磁盘中，称之为 Persist Block

先来看看整体的文件层级：

```
data/
├── 01H2A3B4C5D6E7F8G9H0I1J2K3/  <-- 持久化块 (Block, 目录名为 ULID)
│   ├── chunks/
│   │   ├── 000001               <-- 实际的数据块文件 (压缩后的时间序列数据)
│   │   └── 000002
│   ├── index                    <-- 倒排索引文件 (标签到时间序列的映射)
│   ├── meta.json                <-- 块的元数据 (起止时间、compaction层级等)
│   └── tombstones               <-- 墓碑文件 (记录被删除的数据)
├── 01H2A4B5C6D7E8F9G0H1J2K3L4/  <-- 另一个持久化块
│   └── ...
├── chunks_head/                 <-- Head Block 的内存映射磁盘文件
│   └── 000001                   <-- 刚满的 chunk 切片
├── wal/                         <-- 预写日志 (Write-Ahead Log)
│   ├── 000000001                <-- WAL 段文件
│   ├── 000000002
│   └── ...
└── lock                         <-- 文件锁，防止多实例同时操作同一目录

```

<br />

LSM-Tree 架构的存储引擎中最重要的两个磁盘文件，一个是块文件，存放原始数据，一个是索引文件，存放索引，对应上面的 chunks/ 和 index 两部分。

chunks/ 目录下存放块文件，每个块文件就是 Head Block 数据加上一些元信息。大致格式如下：

```
+-------------------------------------------------------------+
| (Sample 1 ~ Sample 120)                                     |
| ...                                                         |
| [Data End Offset]                                           |
| [Time Range: MinT, MaxT]                                    |
| [CRC32]                                                     |
+-------------------------------------------------------------+

```

<br />

index 文件比较复杂，大致如下组成：

1. **Symbol Table（符号表）**：将所有标签名和标签值（如 `__name__`, `http_requests`, `method="GET"`）去重后存储，并分配唯一 ID。这极大地减少了字符串的冗余存储。
2. **Series Table（序列表）**：记录每个时间序列的标签组合（引用 Symbol ID）以及该序列在 `chunks/` 中的偏移量。
3. **Postings（倒排索引列表）**：从标签指向时间序列 ID 的映射。例如：`method="GET"` -> \[Series ID 1, Series ID 5, Series ID 99...]。
4. **Postings Offset Table（倒排索引偏移表）**：快速定位倒排索引列表在文件中的位置。

我们通过之前的例子来逐级拆解。

**第一层：Symbol Table (符号表)**\
为了节约内存和磁盘，所有的字符串都被去重，分配了一个递增的 ID。

```
Symbol Table:
ID 1: "__name__"
ID 2: "http_requests_total"
ID 3: "method"
ID 4: "GET"
ID 5: "POST"
ID 6: "path"
ID 7: "/api"

```

<br />

**第二层：Series Table (序列表)**\
记录每个时间序列的标签组合以及该序列在 `chunks/` 中的偏移量。

```
Series Table:
--------------------------------------------------------------------
Series ID: 1
  Labels: [(ID 1, ID 2), (ID 3, ID 4), (ID 6, ID 7)] 
          // 即: __name__=http_requests_total, method=GET, path=/api
  Chunks: 
    [Ref: chunks/000001 偏移量 0, 长度 1KB, MinT: 10:00, MaxT: 10:30]
    [Ref: chunks/000001 偏移量 5KB, 长度 1KB, MinT: 10:30, MaxT: 11:00]
    [Ref: chunks/000001 偏移量 9KB, 长度 1KB, MinT: 11:00, MaxT: 11:30]
    [Ref: chunks/000001 偏移量 13KB, 长度 1KB, MinT: 11:30, MaxT: 12:00]
--------------------------------------------------------------------
Series ID: 2
  Labels: [(ID 1, ID 2), (ID 3, ID 5), (ID 6, ID 7)] 
          // 即: __name__=http_requests_total, method=POST, path=/api
  Chunks: 
    [Ref: chunks/000001 偏移量 1KB, 长度 1KB, MinT: 10:00, MaxT: 10:30]
    ...
--------------------------------------------------------------------

```

<br />

有了这张表，如果我知道 Series ID=1，我就能直接去磁盘把它的 4 个 Chunk 读出来。

**第三层：Postings (倒排索引列表)**\
接下来，给定一组标签值，我们如何知道我要找的 Series ID 是多少？

这就需要标签值到 Series ID 的倒排索引，即 Postings。这是实现 `http_requests_total{method="GET"}` 快速查找的核心。Prometheus 会为每一个标签键值对，建立一张倒排索引表，记录含有该标签的所有 Series ID。

```
Postings:
--------------------------------------------------------------------
Label: __name__="http_requests_total" (即 ID 1, ID 2)
  Series List: [1, 2]   <-- 包含此标签的系列有 1 和 2
--------------------------------------------------------------------
Label: method="GET" (即 ID 3, ID 4)
  Series List: [1]      <-- 只有系列 1
--------------------------------------------------------------------
Label: method="POST" (即 ID 3, ID 5)
  Series List: [2]      <-- 只有系列 2
--------------------------------------------------------------------
Label: path="/api" (即 ID 6, ID 7)
  Series List: [1, 2]   <-- 1 和 2 都有
--------------------------------------------------------------------

```

<br />

**第四层：Postings Offset Table (倒排索引偏移表)**\
因为 Postings 列表可能非常长（几百万个 Series ID），为了在磁盘上快速定位，还需要一个“索引的索引”。\
它记录了：`method="GET"` 这个倒排列表，在 Index 文件的具体哪个字节位置。

```
Postings Offset Table:
[Label: ID 3, ID 4] -> Offset: 0x8A00  (指向 method="GET" 的 Postings 起始位置)
[Label: ID 1, ID 2] -> Offset: 0x9000  (指向 __name__ 的 Postings 起始位置)
...

```

<br />

现在有个查询 promQL是 `http_requests_total{method="GET"}`，时间范围是 10:00 - 11:00。

1. **查符号表**：将 `method="GET"` 翻译成符号 ID `(3, 4)`，将 `__name__="http_requests_total"` 翻译为 ID`(1, 2)`
2. **查偏移表**：用 ID `(3, 4)` 在 Postings Offset Table 中找到磁盘偏移量 `0x8A00`，以及 ID `(1, 2)` 的偏移量：`0x9000`
3. **读倒排表**：去 `0x8A00` 读取 Postings，得到 `Series List: [1]`，去 `0x9000` 读取 Postings，得到 `Series List: [1，2]，`取交集为 `Series ID 1`
4. **查序列表**：拿着 `Series ID 1`，去 Series Table 中找到它。
5. **过滤 Chunk**：Series 1 有 4 个 Chunk，对比时间范围 `[10:00, 11:00]`，发现只需要前 2 个 Chunk（MinT 10:00, MaxT 10:30 和 MinT 10:30, MaxT 11:00）。
6. **读取数据**：根据前 2 个 Chunk 的 `Ref (文件名+偏移量)`，去 `chunks/000001` 中直接 `seek` 到对应位置，读取压缩数据。
7. **解压返回**：在内存中用 XOR 和 Delta-of-Delta 算法实时解压，还原出 `[100, 105, 105...]` 返回给查询引擎。

## 3. Query(Calculate)

最后一部分是查询。在上一节我们说了 `http_requests_total{method="GET"}` 背后的查询逻辑，但这只是个最简单的 promQL，查询原始的时间序列，我们一般使用时还要用函数做进一步的操作。

不同的指标类型有不同的处理函数。

Prometheus 虽然有 Gauge Counter Histogram Summary 四种指标类型，但归根到底只有两种，累计值（只增不减）和 瞬时值 （可增可减）。Histogram 本质上是多个分桶的 累计值，Summary 则兼具瞬时值（分位值是Exporter计算出的瞬时值）和累计值。

对于瞬时值类型，只有一些数理统计函数，例如 `avg_over_time()`，计算区间内的平均值，以及 `min_over_time()` `max_over_time()`

对于累计值类型，最核心的是 `rate()` 和 `histogram_quantile()` 两个函数

### 3.1 rate()

对于累计值，看绝对值通常毫无意义，我们需要知道的是某个时间范围的平均每秒增长率。例如：

```
# 计算过去 5 分钟的每秒请求量（QPS）
rate(http_requests_total[5m])
```

<br />

rate() 函数的计算公式是      `  (结束值-起始值) / 时间范围  `      。

计算时需要考虑到 CounterReset，计算公式实际上是 `(结束值-起始值) + 重置前最大值 / 时间范围`。

解释下 CounterReset。Counter 的语义是“只增不减”。但在实际运行中，以下两种情况会导致 Counter 的值突然暴跌归零：

* **进程重启**：这是最常见的场景。一个 Java/Go 进程重启后，其内存中的 `http_requests_total` 从 0 开始重新累计。

* **目标实例发生改变**：比如 Pod 重建，Kubernetes 分配了新的 IP，Prometheus 虽然抓取的是同一个服务名，但底层已经是一个全新的实例，Counter 从 0 开始。

那么 rate() 函数看到突然下降的 counter 值就要用重置前区间最大值进行“补偿”，从而修正最终计算结果。

举个例子，假设一个 Counter 的时序数据如下：

* T1: 100

* T2: 150

* T3: 180

* **T4: 进程重启 -> 10**

* T5: 30

显然计算 T1 到 T5 的增量不能用 `30 - 100`。

我们假设在重置前一刻，Counter 达到了区间最大值，因此在计算增量时需要“补偿”最大值 180，则计算结果为，`30 - 100 + 180 = 110`

<br />

### 3.2 histogram\_quantile()

Histogram 类型的指标在计算分位值时需要用到 `histogram_quantile()`

如下一个示例，总共1000个请求，我们希望找出 `request_duration_seconds` 的 P50 分位值

![1.00](https://pub-2f2f8e6626cf43d389ec1fa8a2d85504.r2.dev/images/1779012477451-puplfy.png)

首先直观上来看，P50 肯定落在 100ms-200ms 之间，我们假设在各个区间 duration 都是线性增大的，那么用下面的线性插值法就能算出来 P50 分位值。

`(785 - 168) / (250 - 100) = (500 - 168) / (x - 100)`

<br />

具体到 promQL，就是 `histogram_quantile(0.50, rate(request_duration_seconds_bucket[5m]))`

我们来拆解一下：

* 在实际计算中，我们一般不会对历史上所有数据求P50，所以会先 `rate(request_duration_seconds_bucket[5m])`，计算过去5min中内各个分桶的增长速率。

* 执行 `histogram_quantile(0.50, ...)` 该函数输入一个分位数 0.5，和一组桶时间序列，然后找到第一个累计分布频率超过50%的桶，然后用线性插值法来计算更精确的值

刚刚我们简化了问题，只有1个实例在上报指标。如果有2个实例呢？我们就需要先将指标数据按照分桶累加，再计算分位值。

![1.00](https://pub-2f2f8e6626cf43d389ec1fa8a2d85504.r2.dev/images/1779017954558-o5qmbx.png)

![1.00](https://pub-2f2f8e6626cf43d389ec1fa8a2d85504.r2.dev/images/1779018076652-04aacu.png)

具体到 promQL，就是 `histogram_quantile(0.50, sum by (le)(rate(request_duration_seconds_bucket[5m])))`

<br />

最后，我们再用这个例子考虑分桶的合理性问题。

如果用上面这个例子计算 P90，可以吗？从图中来看，P90会落在最右边的分桶，桶上边界是 +inf，使用线性插值法返回的结果也是 +inf。

可以看到，**分桶的合理性，直接影响了计算结果的准确性**。桶分的越细，越符合整体分布，计算出来的分位值才越准。

### 3.3 其它聚合函数

`rate()` 和 `histogram_quantile()`**是“语义转换函数”**:

它们的作用是**改变数据的物理意义**，把一种无法直接使用的A，变成可以直观使用的B。

* **`rate()`**：把“累计的绝对值”变成了“瞬时的速率”，从 历史总和到 QPS。

* **`histogram_quantile()`**：把“离散的桶分布”变成了“连续的分位数值”。从一堆 le 标签的计数到 P99延迟。

此外，还有一些不涉及到语义转换的函数，聚合函数，将多个时间序列按照某个维度聚合。例如 ` sum()` `avg()`

<br />

### 3.4 Recording Rule

当一个查询被 Grafana/AlertManager 大量重复使用时，可以使用 Recording Rule 来缓存中间计算结果。

Recording Rule 的核心作用是**预计算**，将经常使用或计算量大的 PromQL 表达式提前算好，并将结果保存为新的时间序列存入TSDB，从而提升查询速度和减轻系统压力。

举个例子，对 `rate(http_requests_total[5m])` 预计算，后续的 `sum(rate(http_requests_total[5m])) `         就可以使用预计算结果。

```
groups:
  - name: http_requests_rules
    rules:
      # 第一步：预计算最细粒度的 rate，保留所有可能用到的标签 (job, pod, instance 等)
      # 注意这里没有用 sum，只是把 rate 算好
      - record: http_requests:rate5m
        expr: rate(http_requests_total[5m])

```
